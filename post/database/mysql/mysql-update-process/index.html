Posted by 松溪 on Tuesday, January 10, 2023

MySQL更新过程

前置过程参考MySQL查询过程

除此之外，重点是以下过程：

  1. 执行器调用InnoDB引擎获取数据。如果数据页在内存中，则直接返回给执行器；否则，先从磁盘读入内存中，再返回。
  2. 执行器获取到数据后，进行相关操作后，调用引擎写入新数据。
  3. 引擎将数据更新到内存，同时将更新操作写到redo log中。此时redo log处于prepare状态，通知执行器执行完成，可以提交事务。
  4. 执行器生成这个操作的binlog，写入磁盘。
  5. 执行器调用引擎的提交事务接口，将redo log改成commit状态，更新完成。
知识点
  • WAL，Write-Ahead Logging。当有一条记录需要更新的时候，InnoDB引擎就会先把记录写到redo log里面，并更新内存，这个时候更新就算完成了。同时，InnoDB 引擎会在适当的时候，将这个操作记录更新到磁盘里面。
  • redo log用于保证crash-safe能力
  • 两阶段提交。确保数据库发生异常时，用日志恢复的状态和异常前的状态保持一致。
binlog和redo log区别
  1. binlog是MySQL的Server 层实现的，所有引擎都可以使用；redo log是InnoDB引擎特有的。
  2. binlog是逻辑日志，记录原始SQL；redo log是物理日志，记录数据页上做了什么修改。
  3. binlog是追加写，文件写到一定大小会切换下一个；redo log是循环写，空间固定会用完。
change buffer

当需要更新一个数据页时，如果数据页在内存中就直接更新，而如果这个数据页还没有在内存中的话，在不影响数据一致性的前提下，InnoDB 会将这些更新操作缓存在 change buffer 中，这样就不需要从磁盘中读入这个数据页了。在下次查询需要访问这个数据页的时候，将数据页读入内存，然后执行 change buffer 中与这个页有关的操作。通过这种方式就能保证这个数据逻辑的正确性。

redo log VS change buffer

MySQL执行更新操作时，存储引擎会将数据更新到内存，同时将更新操作写到 redo log 中。如果数据页不在内存中，会将更新操作缓存在change buffer。

redo log是记录更改操作的必须日志，用于故障恢复，而change buffer是一种优化性能用的缓存结构，它不影响数据库的正常运行，但可以大大减少磁盘I/O操作.

redo log节省了随机写磁盘的IO消耗，转为顺序写；change buffer节省了随机读磁盘的IO消耗。
