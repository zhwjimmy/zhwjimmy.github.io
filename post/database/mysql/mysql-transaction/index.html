MySQL文章系列（五） 事务

Posted by 松溪 on Wednesday, January 11, 2023

事务并发操作的问题

脏读是指一个事务读取到了其他事务未提交的数据。

不可重复读是指事务A查询到数据后，事务B做了修改后提交，此时事务A在此查询数据时发现和前一次数据不一致。

幻读是指事务A查询到数据后，事务B新插入数据，此时事务A再次查询时发现数据变多。

事务的隔离级别

数据库系统中的事务隔离级别是用来保证事务并发操作之间的隔离性的。隔离级别的高低决定了事务之间的可见性和可能的并发冲突。

  • 未提交读
    • 事务 A 可以看到事务 B 未提交的修改
    • 可能会出现脏读、不可重复读、幻读
  • 已提交读（RC）
    • 事务 A 只能看到事务 B 已经提交的修改
    • 这种隔离级别可以避免脏读，但仍然可能会出现不可重复读、幻读
  • 可重复读（RR）
    • 事务执行过程中读到的数据，总是和这个事务启动时看到的数据一致
    • 可以避免脏读和不可重复读，但仍然可能会出现幻读。
  • 串行化
    • 务 A 与事务 B 的操作是完全独立的，类似于串行执行
    • 可以避免脏读、不可重复读、幻读
事务默认的隔离级别

MySQL 默认的事务隔离级别是 REPEATABLE READ，这意味着在同一事务中的所有查询在整个事务过程中都会读取到相同的数据快照，并且其他事务在该事务未提交前不能对读取到的数据进行修改。

这样可以防止脏读、不可重复读问题，但是幻读仍有可能出现。

事务的隔离级别设置为RC

将事务隔离级别设置为 READ COMMITTED 有以下几个优点：

  • 避免脏读问题：在 READ COMMITTED 级别下，事务 A 只能读取到已经提交了的事务 B 的修改，这样就不会出现读取到未提交的数据的情况。
  • 更高的并发能力：在 READ COMMITTED 级别下，多个事务可以同时读取和修改相同的数据，这样可以提高系统的并发能力。
  • 更少的锁等待：由于可以并发读取和修改数据，系统中锁等待的情况会更少，这可以减少系统的延迟。

但是，在使用 READ COMMITTED 级别时也需要注意以下几点：

  • 不可重复读问题可能会出现：因为 READ COMMITTED 级别下的事务可以读取其他事务提交的数据，所以在同一事务中读取两次数据可能会得到不同的结果。
  • 幻读问题可能会出现：因为 READ COMMITTED 级别下的事务可以读取其他事务提交的数据，所以在同一事务中查询两次数据可能会得到不同的结果。
  • 需要注意并发更新问题，需要使用悲观锁或乐观锁来解决。

MVCC

MVCC，Multiversion Concurrency Control，多版本控制。

通过 MVCC 我们可以解决以下几个问题：

  • 读写之间阻塞的问题，通过 MVCC 可以让读写互相不阻塞，即读不阻塞写，写不阻塞读，这样就可以提升事务并发处理能力。
  • 降低了死锁的概率。这是因为 MVCC 采用了乐观锁的方式，读取数据时并不需要加锁，对于写操作，也只锁定必要的行。
  • 解决一致性读的问题。一致性读也被称为快照读，当我们查询数据库在某个时间点的快照时，只能看到这个时间点之前事务提交更新的结果，而不能看到这个时间点之后事务提交的更新结果。
InnoDB 中的 MVCC 实现原理
  1. 每个事务开启时，会向事务系统申请一个唯一的事务ID，叫作transaction id，按申请顺序严格递增。
  2. 数据库中的每行数据是有多个版本的，每次事务更新数据的时候，都会生成一个新的数据版本。每个版本的记录除了数据本身外，还有一个表示版本的字段，记为row trx_id，即对应事务的transaction id。
  3. 每个记录在更新时，会同时记录一条回滚日志，即undo log。通过回滚操作，可以得到前一个状态的值。
  4. 一个事务启动时，会生成当前事务全局性的快照。
    • 实现上，InnoDB为每个事务构造了一个数组，用来保存事务启动时，当前正在活跃的所有事务ID（启动但没提交的事务）。
    • 数组里面事务 ID 的最小值记为低水位，当前系统里面已经创建过的事务 ID 的最大值加 1 记为高水位。
    • 这个视图数组和高水位，就组成了当前事务的一致性视图（read-view）
  5. 数据版本的可见性规则，根据数据的 row trx_id 和这个一致性视图的对比。
    • 数据的row trx_id小于低水位，对当前事务可见
    • 数据的row trx_id大于高水位，对当前事务不可见
    • 数据的row trx_id介于水位之间，取决于row trx_id是否在数组中。在，不可见；不在，可见。
快照读和当前读

所谓快照读，就是读取的是快照数据，即快照生成的那一刻的数据，像我们常用的普通的SELECT语句在不加锁情况下就是快照读。如：

SELECT * FROM xx_table WHERE ...

当前读就是读取最新数据，所以，加锁的 SELECT，或者对数据进行增删改都会进行当前读，比如：

SELECT * FROM xx_table LOCK IN SHARE MODE;
SELECT * FROM xx_table FOR UPDATE;
INSERT INTO xx_table ...
DELETE FROM xx_table ...
UPDATE xx_table ...
RC和RR

RC级别下，快照是SQL语句开始执行的时候生成。

RR级别下，快照是事务开启开始执行的时候生成。

事务隔离级别的实现

未提交读

在 READ UNCOMMITTED 级别下，MySQL 不会使用任何锁来保证事务隔离性。多个事务可以并发读取和修改同一条数据，存在脏读问题。

已提交读

在 READ COMMITTED 级别下，MySQL 会使用读锁来阻塞其他事务对被读取数据的修改操作。

在启动事务时，MySQL 会为该事务创建一个全局的事务 ID，并将该事务 ID 添加到所有访问的行上。当该事务读取数据时，MySQL 会在行上加上读锁，来阻止其他事务对该行进行修改；当该事务修改数据时，MySQL 会在行上加上写锁，来阻止其他事务对该行进行读取和修改。这样可以防止脏读问题。

在 READ COMMITTED 级别下, 当一个事务读取数据时，另一个事务在这个读取操作未提交前不能对数据进行修改。只有读取操作提交后，其它事务才能对数据进行修改。

可重复读

在 REPEATABLE READ 级别下，MySQL使用锁和版本控制来保证事务隔离。

  • 锁。表锁防止了脏读，读锁防止了不可重复读。
    • 表锁: MySQL 会在事务读取数据时，对访问的表加上表锁，阻止其他事务对该表进行修改。这样可以保证在同一事务中多次读取同一数据时，数据不会被其他事务修改。
    • 读锁: MySQL 会在事务读取数据时，在数据上加上读锁。这样可以阻止其他事务修改或删除这条数据，并同时允许其他事务读取这条数据。
  • 版本控制。MVCC，Multi-Version Concurrency Control，多版本并发控制，一种并发控制的方法。。它允许多个事务并发读取同一数据，同时保证数据的一致性。MVCC 的实现原理是在每一条数据上维护一个版本号，每当事务修改该条数据时，就会创建一个新版本。读取数据时，MySQL 会根据事务的版本号来返回对应版本的数据。这样同一事务的多次读取操作都返回相同的结果，这就是可重复读的实现
串行化

在 REPEATABLE READ 和 SERIALIZABLE 级别下，MySQL 会使用额外的锁来保证事务隔离性，避免了不可重复读和幻读问题，但带来了更高的开销。

值得注意的是，在 InnoDB 存储引擎中，默认使用 REPEATABLE READ 级别来保证事务隔离性。

在 REPEATABLE READ 级别下，MySQL 会使用读锁来阻塞其他事务对被读取数据的修改操作，并在访问数据之前使用读锁锁定所有访问的表。

在 SERIALIZABLE 级别下，MySQL 会使用读锁来阻塞其他事务对被读取数据的修改操作，并在访问数据之前使用写锁锁定所有访问的表。

MySQL 也支持通过使用乐观锁的方式来解决并发问题。它提供了 SELECT … FOR UPDATE 和 SELECT … LOCK IN SHARE MODE 两种方式来获取行级锁，以避免并发问题。

需要注意，MySQL 使用锁来实现事务隔离级别会导致性能问题，特别是在并发性较高的应用场景中，需要合理使用事务隔离级别和锁来避免性能问题。
