MySQL文章系列（五） 事务

Posted by 松溪 on Wednesday, January 11, 2023
Last Modified on Friday, January 13, 2023

事务并发操作的问题

  • 脏读。指一个事务读取到了其他事务未提交的数据。
  • 不可重复读。指事务A查询到数据后，事务B做了修改后提交，此时事务A在此查询数据时发现和前一次数据不一致。
  • 幻读。指事务A查询到数据后，事务B新插入数据，此时事务A再次查询时发现数据变多。

事务的隔离级别

数据库系统中的事务隔离级别是用来保证事务并发操作之间的隔离性的。隔离级别的高低决定了事务之间的可见性和可能的并发冲突。

  • 未提交读
    • 事务 A 可以看到事务 B 未提交的修改
    • 可能会出现脏读、不可重复读、幻读
  • 已提交读（RC）
    • 事务 A 只能看到事务 B 已经提交的修改
    • 这种隔离级别可以避免脏读，但仍然可能会出现不可重复读、幻读
  • 可重复读（RR）
    • 事务执行过程中读到的数据，总是和这个事务启动时看到的数据一致
    • 可以避免脏读和不可重复读，但仍然可能会出现幻读。
    • 默认的事务隔离级别。
  • 串行化
    • 务 A 与事务 B 的操作是完全独立的，类似于串行执行
    • 可以避免脏读、不可重复读、幻读

MVCC

MVCC，Multiversion Concurrency Control，多版本控制。是InnoDB存储引擎提供的一种并发控制机制。

并发场景下，可能会出现三种情况：

  • 读-读并发
  • 读-写并发
  • 写-写并发

读-读并发不会出问题，写-写并发场景主要通过加锁解决。而读-写并发场景，除了加锁（共享锁/排它锁）外，还可以通过MVCC机制解决。

MVCC机制使得InnoDB引擎不再单纯的使用行锁来进行数据库的并发控制，取而代之的是把数据库的行锁与行的多个版本结合起来，只需要很小的开销,就可以实现非锁定读，从而大大提高数据库系统的并发性能。

解决并发问题的方案有：

  • 对并发数据加排他锁，其他的读、写操作都需要等到锁释放后才能操作
  • 对并发数据加共享锁，可以正常读，但写操作阻塞
  • MVCC机制，对数据进行多版本控制，避免因写操作而导致的读操作的阻塞
InnoDB 中的 MVCC 实现原理

MVCC的两个实现核心是undo log和一致性视图，通过undo log来保存多版本的数据，通过一致性视图来保存当前活跃的事务列表，将两者结合和制定一定的规则来判断当前可读数据。

  1. 事务版本号。每个事务开启时，会向事务系统申请一个唯一的事务ID，叫作transaction id，按申请顺序严格递增。
  2. 隐式字段。数据库中的每行数据是有多个版本的，每次事务更新数据的时候，都会生成一个新的数据版本。每个版本的记录除了数据本身外，还有一个表示版本的字段，记为row trx_id，即对应事务的transaction id。
  3. undo log（或叫undo log 版本链）。每个记录在更新时，会同时记录一条回滚日志，即undo log。通过回滚操作，可以得到前一个状态的值。
  4. 一致性视图（read-view）。事务启动时，数据库会为当前事务创建一个快照（一致性视图）。
    • 实现上，InnoDB为每个事务构造了一个数组，用来保存事务启动时，当前正在活跃的所有事务ID（启动但没提交的事务）。
    • 数组里面事务 ID 的最小值记为低水位，当前系统里面已经创建过的事务 ID 的最大值加 1 记为高水位。
    • 这个视图数组和高水位，就组成了当前事务的一致性视图（read-view）
  5. 可见性判断。根据数据的 row trx_id 和一致性视图（read-view），获取对应的数据版本。
    • 数据的row trx_id小于低水位，对当前事务可见
    • 数据的row trx_id大于高水位，对当前事务不可见
    • 数据的row trx_id介于水位之间，取决于row trx_id是否在数组中。在，不可见；不在，可见。
快照读

所谓快照读，就是读取的是快照数据，即快照生成的那一刻的数据，像我们常用的普通的SELECT语句在不加锁情况下就是快照读。如：

SELECT * FROM xx_table WHERE ...

个人的一些思考和总结：关于快照读和MVCC关系，查了很多资料，但相互矛盾，有的说快照读是基于MVCC实现的，有的说MVCC是基于快照实现的。个人理解，MVCC是解决并发场景中读-写问题的一种机制，最终达成的效果就是另一个事务修改某一行数据时，当前事务读取不受影响，那么这种情况就叫快照读。而当前读，就和MVCC无关了，是通过加锁实现的。

RC级别下，快照是SQL语句开始执行的时候生成。所以，快照读和当前读的数据是一直的。

RR级别下，快照是事务开启开始执行的时候生成。

当前读

当前读就是读取最新数据，所以，加锁的 SELECT，或者对数据进行增删改都会进行当前读，比如：

SELECT * FROM xx_table LOCK IN SHARE MODE;
SELECT * FROM xx_table FOR UPDATE;
INSERT INTO xx_table ...
DELETE FROM xx_table ...
UPDATE xx_table ...

Mysql实现当前读是通过共享锁+排他锁+Next-Key Lock实现的。

  • 每次对行数据进行读取的时候，加共享锁。此时就不允许修改，但是允许其他事务读取，所以每次都可以读到最新的数据。
  • 每次对行数据进行修改的时候，加排他锁，不允许其他事务读取和修改。这种情况下其他事务读取的数据也一定是最新的数据。
  • 每次对范围行数据进行读取的时候，对这个范围加一个范围共享锁。
  • 每次对范围行数据进行修改的时候，读这个范围加一个范围排它锁。

事务默认的隔离级别

MySQL 默认的事务隔离级别是 REPEATABLE READ，这意味着在同一事务中的所有查询在整个事务过程中都会读取到相同的数据快照，并且其他事务在该事务未提交前不能对读取到的数据进行修改。

这样可以防止脏读、不可重复读问题，但是幻读仍有可能出现。

事务的隔离级别设置为RC

将事务隔离级别设置为 READ COMMITTED 有以下几个优点：

  • 避免脏读问题：在 READ COMMITTED 级别下，事务 A 只能读取到已经提交了的事务 B 的修改，这样就不会出现读取到未提交的数据的情况。
  • 更高的并发能力：在 READ COMMITTED 级别下，多个事务可以同时读取和修改相同的数据，这样可以提高系统的并发能力。
  • 更少的锁等待：由于可以并发读取和修改数据，系统中锁等待的情况会更少，这可以减少系统的延迟。

但是，在使用 READ COMMITTED 级别时也需要注意以下几点：

  • 不可重复读问题可能会出现：因为 READ COMMITTED 级别下的事务可以读取其他事务提交的数据，所以在同一事务中读取两次数据可能会得到不同的结果。
  • 幻读问题可能会出现：因为 READ COMMITTED 级别下的事务可以读取其他事务提交的数据，所以在同一事务中查询两次数据可能会得到不同的结果。
  • 需要注意并发更新问题，需要使用悲观锁或乐观锁来解决。

事务隔离级别的实现

未提交读

在 READ UNCOMMITTED 级别下，MySQL 不会使用任何锁来保证事务隔离性。多个事务可以并发读取和修改同一条数据，存在脏读问题。

已提交读

读操作，通过MVCC机制实现。但可见性判断是基于语句级别的ReadView。因此不可重复读。

写操作，只有记录锁，没有间隙锁。

可重复读

读不加锁，写加锁，读写互不阻塞，并发度相对于串行化要高，但会有幻读的发生。

读操作，通过MVCC机制实现。 写操作，加锁。包括行锁，间隙锁，表锁。

串行化

执行两阶段锁协议，事务访问到的时候加锁，事务提交的时候释放锁。读写都要加锁。
