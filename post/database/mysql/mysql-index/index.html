MySQL文章系列（三） 索引

Posted by 松溪 on Wednesday, January 11, 2023

索引分类

按数据结构分类
  • B+tree索引
    • 基于B+tree索引的索引类型有主键索引、唯一索引、普通索引
  • Hash索引
  • 全文索引
按物理存储分类
  • 聚簇索引（InnoDB中，主键索引为聚簇索引）
    • 叶子节点存放的是整行数据
  • 二级索引
MyISAM vs InnoDB

MyISAM 和 InnoDB 是MySQL中两种常用的存储引擎。主要区别如下：

  1. 数据存储结构的区别。
  • MyISAM
    • 数据文件（.MYD）: 存储数据表中所有记录的数据，采用二进制形式存储。
    • 索引文件（.MYI）: 存储数据表中所有索引的数据，采用B+树或Hash索引组织数据。
    • 表信息文件（.frm）: 存储数据表的结构信息，包括表的列信息、类型、长度、约束等。
  • InnoDB
    • 数据文件（.ibd）: 存储数据表中所有记录的数据，采用二进制形式存储。
    • 索引文件（.ibd）: 存储数据表中所有索引的数据，采用B+树组织数据。
    • 表信息文件（.frm）: 存储数据表的结构信息，包括表的列信息、类型、长度、约束等。
  1. 索引的区别。InnoDB支持聚簇索引，MyISAM不支持；InnoDB在索引上只支持B+树，而MyISAM支持B+树和Hash索引；
  2. 锁的区别。InnoDB支持行锁，MyISAM只支持表锁；
  3. 事务的区别。InnoDB支持事务，支持多版本并发控制(MVCC)，MyISAM不支持；
  4. InnoDB支持外键，MyISAM不支持；
普通索引 vs 唯一索引

普通索引和唯一索引是不同类型的索引，有着不同的特性和用途。

  • 普通索引是一种普通的索引，字段上可以存在重复的值，主要用于优化查询性能；
  • 唯一索引是一种特殊的所有，索引值在表中要求是唯一的，主要用于保证表中某一列的数据不重复。

普通索引的建立方式比较简单，只需要使用CREATE INDEX语句就可以建立，而唯一索引需要在建立索引时加上UNIQUE关键字。

对于查询过程，普通索引查到满足条件的记录后，需要往后在查直到碰到不满足条件的记录；而唯一索引由于记录唯一，查到第一个满足条件的记录就会停止检索。由于InnoDB的数据是按数据页为单位读写的，查询时会将整页读入内存，因此二者的性能差距微乎其微。

对于更新过程，如果要更新的目标页在内存中，二者的操作基本一致；但如果目标页不在内存中，对于唯一索引，需要把数据页读入内存，判断到没有冲突，插入这个值，语句执行结束；而对于普通索引，将更新记录在change buffer，语句执行就结束了。

主键索引 vs 普通索引

主键索引是表中唯一标识一条记录的索引，它需要具有唯一性和非空性。每个表只能有一个主键索引，主键索引可以被用来联接其他表。

唯一索引是表中唯一标识一条记录的索引，但是它不一定是主键索引。每个表可以有多个唯一索引，但是每个索引的列值都必须是唯一的。

B 树 vs B+ 树

B树和B+树都属于多路平衡查找树，保证了查询和修改操作的时间复杂度为对数级别；都是用于磁盘或者其他外存设备的数据组织结构。主要区别如下：

  1. B数的每个节点都可能存储数据；而B+树只有叶子节点存储数据，非叶子节点只存储索引。
    1. B树的空间利用率较高，但非叶子节点能保存的指针数量变小，所以要要保存大量数据，只能增加树的高度，导致 IO 操作变多，查询性能变低。
  2. B+树的叶子节点采用双向链表
  3. B树适用于磁盘或其他外存设备上的数据存储，而B+树更适用于索引数据库和文件系统中的数据存储.
B+ 树 vs 跳表

B+ 树和跳表都是用于大量数据中提升查询效率的数据结构，查询的时间复杂度为对数级。主要区别如下：

  1. B+树是多路平衡查找树，跳表是链表结构。
  2. B+树主要适合外部存储，可以减少磁盘IO数；跳表比较适合内存存储。

索引常见概念

  • 回表
  • 覆盖索引
    • 可以在非主键索引上查到相关数据，不需要回表查询
  • 最左前缀原则
    • 可以是联合索引的最左 N 个字段，也可以是字符串索引的最左 M 个字符
  • 索引下推
    • MySQL5.6中，对索引中包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数
    • select * from tuser where name like ‘张%’ and age=10 and ismale=1

常见场景

为什么MySQL会选错索引

选择索引是优化器的工作，而优化器会选择一个它认为最优的执行计划。最优的执行计划一般由扫描的行数，临时表及排序等因素决定。

  1. 索引基数错误

MySQL通过索引的基数来预估需要扫描的行数。该数是一个估计值，通过采样统计计算索引的基数。 InnoDB 默认会选择 N 个数据页，统计这些页面上的不同值，得到一个平均值，然后乘以这个索引的页面数，就得到了这个索引的基数。

解决方案：重新统计索引信息，analyze table t

  1. 复杂语句选错索引

解决方案：

  • 采用force index强行选择一个索引
  • 通过修改语句，引导MySQL使用期望的索引
  • 新建一个更合适的索引，或者删除误用的索引
