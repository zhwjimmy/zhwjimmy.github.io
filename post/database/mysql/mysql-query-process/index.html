MySQL查询过程

首先，客户端和MySQL服务器建立连接。连接器负责和客户端建立连接、获取权限和管理连接。

第二步，MySQL服务器接收到请求后，先检查查询缓存。如果命中了缓存，则立即返回存储在缓存中的结果。否则进入下一阶段。但使用查询缓存往往弊大于利，因为查询缓存的失效非常频繁，只要表更新，表相关的查询缓存都会被清空，所以更新频率比较高的话，查询缓存的命中率会非常低。

下边开始真正执行语句。

第三步，解析阶段。MySQL分析器进行词法分析、语法分析，如果语法正确，MySQL会将该语句解析生成解析树；如果语法错误，MySQL会返回错误信息。

第四步，优化阶段。MySQL优化器会根据语句内容、数据库的统计信息和索引的使用情况等来确定最优执行计划，包括使用哪个索引、如何进行连接、如何对结果排序等。

第五步，执行阶段。MySQL执行器会根据生成的执行计划，操作引擎提供的接口来执行查询，将结果返回给客户端。

排序

MySQL查询优化

  1. 具体层面，SQL语句优化
    1. 尽可能的使用到索引，避免全表扫描
    2. 索引分普通索引、唯一索引、联合索引和主键索引等，建表时根据查询情况创建合适的索引
    3. 联合索引的最左前缀原则
    4. 索引从物理存储上分为聚簇索引和非聚簇索引，非聚簇索引叶子节点存储的是索引，所以需要回表查询，但如果是覆盖索引时，则不需要再回表
    5. 尽量避免select *，只查询需要的字段
    6. 有些情况是不走索引的，需要注意
      1. 索引列字段参与计算，或者使用了函数
      2. 使用了like模糊查询
      3. 字符串和数字进行比较
      4. 不等于查询
      5. null查询
    7. 进行SQL语句优化时，可以使用explain语句
    8. 通过慢查询语句，监测执行耗时长的语句，进行分析优化
  2. 建表
    1. 遵循三范式
    2. 选择合适的字段、长度，建立合适的索引
    3. 字段尽量非空
    4. 单表字段不易过多
  3. 数据库层面
    1. 选择合适的存储引擎
    2. 表过大要进行分库分表
    3. 使用redis等减轻查询压力
    4. 连接池
    5. 集群部署，读写分离

慢查询分析

explain分析

  • id 表示查询语句的执行顺序
    • id值越大，优先级越高，越先执行
  • select_type 表示查询的类型
    • simple 简单查询
    • primary 复杂语句中，最外层查询
    • subquery 子查询语句
    • union
  • table 表名
  • partitions 分区信息
  • type 查询中使用了何种类型
    • all 全表扫描
    • index 遍历索引树、range 范围查找
    • ref 使用了普通索引
    • eq_ref 使用主键或者唯一索引，用于连表查询
    • const 使用主键或者唯一索引
    • system 表中仅有一行记录
  • possible_keys
  • key 查询中实际使用到的索引
  • key_len 查询中用到的索引长度，原则上越短越好
  • rows 需要扫描的行数
  • extra
    • 覆盖索引
    • 临时表
    • filesort

为什么MySQL会选错索引

介绍mysql的架构、查询过程

索引选择是在优化器阶段。优化器会根据需要扫描的行数、临时表、排序等因素选择一个它认为最优的执行计划。

其中需要扫描的行数是通过索引的基数来预估。索引的基数是一个预估值，是通过采样统计得到，所以可能是一个错误的值，从而导致预估的扫描行数错误。（InnoDB 默认会选择 N 个数据页，统计这些页面上的不同值，得到一个平均值，然后乘以这个索引的页面数，就得到了这个索引的基数。解决方案：重新统计索引信息，analyze table t ）

此外还有可能是：

  • 查询的行数比较多，优化器认为使用索引价值不高；
  • SQL特殊的操作没有使用索引；
  • 使用了复杂的条件组合，导致优化器无法有效的选择索引。

解决方案：

  • 采用force index强行选择一个索引
  • 通过修改语句，引导MySQL使用期望的索引
  • 新建一个更合适的索引，或者删除误用的索引

选择索引是优化器的工作，而优化器会选择一个它认为最优的执行计划。最优的执行计划一般由扫描的行数，临时表及排序等因素决定。

统计总行数COUNT(*)

  • 不同存储引擎
    • MyISAM引擎把一个表的总行数存储在磁盘，直接返回。加where条件失效。
    • InnoDB逐行计算
  • InnoDB为什么不存起来
    • 即使是在同一个时刻的多个查询，由于多版本并发控制（MVCC）的原因，InnoDB 表“应该返回多少行”也是不确定的。
  • count(字段) < count(主键 id) < count(1) ≈ count(*)
    • count(字段)，InnoDB会把每行的字段值取出来，返回给server层。server层拿到字段值后，按行累加
    • count(主键 id) ，InnoDB会把每行的id值取出来，返回给server层。server层拿到id后，判断不可能为空的，按行累加
    • count(1)，InnoDB不取值。server层对返回的每一行，放1进去，按行累加
    • count(*)，InnoDB不取值。server层对返回的每一行，按行累加

数据页

  • InnoDB的数据是按页存储
  • 删除某个记录，只是把记录标记为删除
  • 删除数据页的所有记录，整个数据页可以被复用
  • 删除整个表的数据，所有的数据页标记为可复用，但磁盘上的文件不会变小
  • 按索引递增顺序插入数据，索引是紧凑的；如果数据是随机插入的，可能造成数据页分裂

内存数据页

  • MySQL的内存数据页是在Buffer Pool（BP）中管理的
  • Buffer Pool作用
    • WAL机制中，起到加速更新作用，避免随机写盘
    • 加速查询。直接从内存中查询
  • 内存淘汰策略
    • LRU算法
  • 改进
    • 在 InnoDB 实现上，按照 5:3 的比例把整个 LRU 链表分成了 young 区域和 old 区域
    • 新插入的数据页，放在old处
    • 处于old区域的数据页，每次访问时作判断。若在链表中存在的时间超过1秒，移到链表头部；若短于1秒，不做处理。
