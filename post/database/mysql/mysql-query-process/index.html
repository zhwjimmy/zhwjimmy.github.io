MySQL文章系列（一） 查询过程

Posted by 松溪 on Tuesday, January 10, 2023
Last Modified on Friday, January 13, 2023

MySQL逻辑架构

  • 网络连接层
    • 与MySQL服务器建立连接
  • 服务层（MySQL Server）
    • 连接池
    • 查询缓存
      • 以key-value对的形式缓存在内存中，命中缓存则直接返回
      • MySQL8.0版本已移除查询缓存功能
    • 分析器
      • 词法分析，语法分析（SQL解析生成一个解析树）
    • 优化器
      • 选择索引，决定表的连接顺序，生成执行计划
    • 执行器
      • 判断是否有表的执行权限，操作引擎提供的接口，返回结果
  • 存储引擎层
  • 系统文件层
    • 文件的物理存储层

MySQL查询过程

首先，客户端和MySQL服务器建立连接。连接器负责和客户端建立连接、获取权限和管理连接。

第二步，MySQL服务器接收到请求后，先检查查询缓存。如果命中了缓存，则立即返回存储在缓存中的结果。否则进入下一阶段。但使用查询缓存往往弊大于利，因为查询缓存的失效非常频繁，只要表更新，表相关的查询缓存都会被清空，所以更新频率比较高的话，查询缓存的命中率会非常低。

下边开始真正执行语句。

第三步，解析阶段。MySQL分析器进行词法分析、语法分析，如果语法正确，MySQL会将该语句解析生成解析树；如果语法错误，MySQL会返回错误信息。

第四步，优化阶段。MySQL优化器会根据语句内容、数据库的统计信息和索引的使用情况等来确定最优执行计划，包括使用哪个索引、如何进行连接、如何对结果排序等。

第五步，执行阶段。MySQL执行器会根据生成的执行计划，操作引擎提供的接口来执行查询，将结果返回给客户端。

数据页

  • nnoDB的数据是按页存储
  • 删除某个记录，只是把记录标记为删除
  • 删除数据页的所有记录，整个数据页可以被复用
  • 删除整个表的数据，所有的数据页标记为可复用，但磁盘上的文件不会变小
  • 按索引递增顺序插入数据，索引是紧凑的；如果数据是随机插入的，可能造成数据页分裂

内存数据页

  • MySQL的内存数据页是在Buffer Pool（BP）中管理的
  • Buffer Pool作用
    • WAL机制中，起到加速更新作用，避免随机写盘
    • 加速查询。直接从内存中查询
  • 内存淘汰策略
    • LRU算法
  • 改进
    • 在 InnoDB 实现上，按照 5:3 的比例把整个 LRU 链表分成了 young 区域和 old 区域
    • 新插入的数据页，放在old处
    • 处于old区域的数据页，每次访问时作判断。若在链表中存在的时间超过1秒，移到链表头部；若短于1秒，不做处理。

统计总行数COUNT(*)
  • 不同存储引擎
    • MyISAM引擎把一个表的总行数存储在磁盘，直接返回。加where条件失效。
    • InnoDB逐行计算
  • InnoDB为什么不存起来
    • 即使是在同一个时刻的多个查询，由于多版本并发控制（MVCC）的原因，InnoDB 表“应该返回多少行”也是不确定的。
  • count(字段) < count(主键 id) < count(1) ≈ count(*)
    • count(字段)，InnoDB会把每行的字段值取出来，返回给server层。server层拿到字段值后，按行累加
    • count(主键 id) ，InnoDB会把每行的id值取出来，返回给server层。server层拿到id后，判断不可能为空的，按行累加
    • count(1)，InnoDB不取值。server层对返回的每一行，放1进去，按行累加
    • count(*)，InnoDB不取值。server层对返回的每一行，按行累加
trancate、delete和drop的区别
  • 在速度上，一般来说，drop > truncate > delete
  • truncate 和 delete 只删除数据不删除表的结构；drop语句将数据和表结构全部删除，依赖于该表的存储过程、函数将保留，但变为invalid状态
  • truncate自增字段从头开始计数了，而delete的仍保留原来的数值
