Redis

Posted by 松溪 on Monday, February 6, 2023
Last Modified on Tuesday, February 7, 2023

Redis是一种支持key-value等多种数据结构的存储系统。可用于缓存，事件发布或订阅，高速队列等场景。支持网络，提供字符串，哈希，列表，队列，集合结构直接存取，基于内存，可持久化。

redis为什么快

  • 内存数据库，绝大部分操作都在内存中完成
  • 高效的数据结构
  • 单线程模型，避免了频繁的上下文切换导致的CPU消耗，以及可能的锁竞争问题
  • 使用基于IO多路复用机制的线程模型
  • 很多环节使用异步机制，如持久化、主从复制，避免了单线程模型的阻塞

键值对组织方式

  • redis使用了一个哈希表来保存所有的键值对
  • 每个哈希桶中的entry元素了key和value指针，分别指向了实际的键和值
  • 指针是redisObject结构，包括8字节的元数据和8字节的指针

功能应用

分布式锁

使用setnx命令，设置键值对。如果键不存在，则设置成功；否则设置失败，返回0；使用lua脚本释放锁。

注意事项

  • 加锁的时候要设置一个过期时间，且必须是原子性的操作。避免客户端发生意外导致锁一直未释放
  • 锁变量必须是唯一值，避免释放非自己持有的锁
  • 释放锁的时候使用lua脚本，保证一致性

缺陷：redis实例故障，锁就不存在了

基于多个Redis节点实现高可靠的分布式锁-Redlock

  • 让客户端和多个独立的Redis实例依次请求加锁，如果客户端能够和半数以上的实例成功地完成加锁操作，则认为客户端成功的获得分布式锁
  • 释放锁，执行Lua脚本

内存管理

redis使用内存分配器分配内存。内存分配器一般是按固定大小来分配内存，而不是完全按照应用程序申请的内存空间大小给程序分配。

  • 惰性过期
  • 定期过期
    • 扫描一定数量的数据库的expires字典中一定数量的key，并清除其中已过期的key

  • 不进行数据淘汰。内存满了后新写入会报错
  • 针对全部数据

    • 随机淘汰

LRU淘汰

  • 核心思想
    • 如果一个数据刚刚被访问，那么这个数据肯定是热数据，还会被再次访问
  • 实现
    • LRU算法需要双向链表记录数据的最近访问顺序，一方面有额外的空间开销，另一方面大量数据被访问会带来很多链表操作，影响性能。因此Redis做了简化。
    • 在每个数据对应的 RedisObject 结构体中设置一个 lru 字段，用来记录数据的访问时间戳。
    • Redis 在决定淘汰的数据时，第一次会随机选出 N 个数据，把它们作为一个候选集合。接下来，Redis 会比较这 N 个数据的 lru 字段，把 lru 字段值最小的数据从缓存中淘汰出去。
    • 当需要再次淘汰数据时，Redis 需要挑选数据进入第一次淘汰时创建的候选集合。这儿的挑选标准是：能进入候选集合的数据的 lru 字段值必须小于候选集合中最小的 lru 值。当有新数据进入候选数据集后，如果候选数据集中的数据个数达到了 maxmemory-samples，Redis 就把候选数据集中 lru 字段值最小的数据淘汰出去。
  • 弊端
    • 在处理扫描式单次查询操作时，无法解决缓存污染
  • 解决方案
    • LFU策略，根据数据访问的时效性和被访问次数两个维度来筛选并淘汰数据

LFU淘汰

  • 概念
    • 在 LRU 策略基础上，为每个数据增加了一个计数器，来统计这个数据的访问次数。当使用 LFU 策略筛选淘汰数据时，首先会根据数据的访问次数进行筛选，把访问次数最低的数据淘汰出缓存。如果两个数据的访问次数相同，LFU 策略再比较这两个数据的访问时效性，把距离上一次访问时间更久的数据淘汰出缓存。
  • 实现
    • Redis 是用 RedisObject 结构来保存数据的，RedisObject 结构中设置了一个 lru 字段，用来记录数据的访问时间戳
    • Redis 在实现 LFU 策略的时候，只是把原来 24bit 大小的 lru 字段，又进一步拆分成了两部分 ■ 前16位，保存数据的访问时间戳 ■ 后8位，保存数据的访问次数
    • 当 LFU 策略筛选数据时，Redis 会在候选集合中，根据数据 lru 字段的后 8bit 选择访问次数最少的数据进行淘汰。当访问次数相同时，再根据 lru 字段的前 16bit 值大小，选择访问时间最久远的数据进行淘汰。
  • 计数规则
    • 每当数据被访问一次时，首先，用计数器当前的值乘以配置项 lfu_log_factor 再加 1，再取其倒数，得到一个 p 值；然后，把这个 p 值和一个取值范围在（0，1）间的随机数 r 值比大小，只有 p 值大于 r 值时，计数器才加 1。

读写穿透模式，Read-Through/Write-Through

服务端把缓存作为主要数据存储。应用程序跟数据库缓存交互，都是通过抽象缓存层完成的。

和Cache-Aside相比，就是多了一层Cache-Provider

异步缓存写入，Write behind

同样由Cache Provider来负责缓存和数据库的读写。但不同之处在于，上一个是同步更新缓存和数据，这个则只更新缓存，不更新数据库，通过批量异步的方式更新数据库。

需要考虑的几个问题
  • 删除缓存，而非更新缓存
  • 先更新数据库，后删除缓存
  • 读取binlog异步删除缓存

缓存雪崩

某一时刻大量缓存失效，导致大量的请求访问数据库。

解决方案：

  • 打散key的过期时间
  • 服务降级，暂停非核心业务访问缓存和数据库

缓存击穿

热点缓存数据过期，导致大量请求直接访问数据库。

解决方案：热点数据不设置过期时间。

缓存穿透

缓存和数据库中都没有请求的数据。比如，数据被误删除，或者恶意攻击。

解决方案：缓存空值或缺省值；布隆过滤器；前端进行参数校验。

缓存污染

访问频次非常低的数据留在缓存中，占用缓存空间。导致缓存写入新数据时，需要先做数据淘汰，引入额外的操作开销，影响性能。

使用LFU内存淘汰策略。

AOF日志

将redis的写操作记录到日志中。

三种配置：Always、EverySec、No

AOF重写机制。重写过程由子进程完成。

AOF性能问题

AOF是在命令执行后才会记录日志，因此不会阻塞当前的写操作。

两个潜在风险：

  • 命令执行完还未记录日志时宕机
  • AOF日志是在主线程中执行，如果写入日志很大，可能会阻塞下一个写操作
备份过程中写的问题

一个拷贝，两处日志。

RDB内存快照

bgsave，创建一个子进程，专门用于写入RDB文件，避免主线程的阻塞。

RDB性能问题

fork过程会阻塞主线程。内存越大，阻塞时间越长。

Redis 4.0 中提出了一个混合使用 AOF 日志和内存快照的方法。简单来说，内存快照以一定的频率执行，在两次快照之间，使用 AOF 日志记录这期间的所有命令操作。

主从同步

https://time.geekbang.org/column/article/272852

主从同步的方式

  1. replicaof（Redis 5.0 之前使用 slaveof）命令形成主库和从库的关系
  2. 主从库之间建立连接，协商同步
  3. 主库将所有数据同步给从库，从库收到数据后，在本地完成数据加载 a. 主库执行bgsave命令，生成RDB文件，将文件发送给从库。(在主库将数据同步给从库的过程中，主库不会被阻塞，仍然可以正常接收请求。但是，这些请求中的写操作并没有记录到刚刚生成的 RDB 文件中。为了保证主从库的数据一致性，主库会在内存中用专门的 replication buffer，记录 RDB 文件生成后收到的所有写操作) b. 从库收到RDB文件后，先清空当前数据库，然后加载RDB文件
  4. 主库将新写命令发送给从库

主从库建网络中断

  • 当主从库断连后，主库会把断连期间收到的写操作命令，写入 replication buffer，同时也会把这些操作命令也写入 repl_backlog_buffer 这个缓冲区。
    • repl_backlog_buffer 是一个环形缓冲区，主库会记录自己写到的位置，从库则会记录自己已经读到的位置。
  • 主从库的连接恢复之后，从库会给主库发送 psync 命令。主库把master_repl_offset 和 slave_repl_offset 之间的命令操作同步给从库。

主从数据不一致

  • 原因：主从库间的同步是异步进行，可能会有延迟。
    • 网络延迟
    • 同步命令被阻塞
  • 解决方式
    • 硬件、网络环境
    • 监控

哨兵机制

实现主从库自动切换

哨兵机制的基本流程

  • 监控
    • 哨兵进程周期性地给所有实例发送PING命令，检测是否在运行。如果在规定时间内没有响应，则标记为下线状态
    • 核心是判断主库的下线状态。
      • 分为主观下线和客观下线
      • 当有 N 个哨兵实例时，最好要有 N/2 + 1 个实例判断主库为“主观下线”，才能最终判定主库为“客观下线”。
  • 选主
    • 按照一定的规则从多个从库中选择一个作为新的主库
      • 筛选 + 打分
  • 通知
    • 将新主库的信息发送给其他从库和客户端

哨兵集群

  • 基于pub/sub机制，哨兵实例之间实现相互发现
  • 哨兵通过向主库发送INFO命令，获取从库列表信息

数据分片

数据扩展模式

  • 纵向扩展
    • 实施起来比较简单
    • 持久化和主从复制数据量会很大，耗时会变长
    • 纵向扩展会受到硬件和成本的限制
  • 横向扩展

实现原理

节点取余
  • 优点：规则简单
  • 缺点：增加、减少节点，会导致大部分缓存失效
  • 技巧：多倍扩容
一致性哈希
  • 将节点进行哈希，映射到一个0~2^32的哈希环上
  • 将数据进行哈希，也映射到哈希环上，将映射的结果值顺时针方向找到的第一个节点，就是存储该数据的节点
  • 优化：针对数据倾斜问题，增加虚拟节点
哈希槽
  • Redis Cluste采用哈希槽，来实现数据和实例之间的映射关系
  • 一个切片集群共有16384个哈希槽，通过cluster create命令创建集群时，会将槽点平均分布在集群实例上
  • 每个key根据crc16算法，映射到一个哈希槽中
  • 虽然没有一致性哈希灵活，但实现简单，节点增减也很方便
客户端如何定位数据
  • Redis实例会把自己的哈希槽信息发给其他实例，这样每个实例都有全部哈希槽的映射关系
  • 客户端和实例建立连接后，实例会把哈希槽的分配信息发给客户端
  • 重定向机制。集群提供重定向机制，返回新实例的访问地址
为什么是16384个槽
  • crc16算法产生的哈希值有16bit，可以产生2^16=65536个值
  • 集群实例之间会发送心跳包，如果槽位有65536个，消息头会很大，浪费带宽
  • redis作者认为不建议节点数量超过1000个，节点数过多会导致网络拥堵。所以16384个槽点也够用了
  • redis的哈希槽信息是通过bitmap保持的，传输过程汇总会进行压缩，槽位少，节点少的情况，压缩比例高
