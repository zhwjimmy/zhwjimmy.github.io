消息队列

消息队列的作用

  • 应用解耦
  • 削峰填谷
  • 异步处理

消息队列产品

  • RabbitMQ： RabbitMQ 是一款开源的 AMQP（高级消息队列协议）消息队列中间件，由 Erlang 语言编写。
  • Apache Kafka： Kafka 是一款高吞吐量、低延迟的分布式消息队列系统。它支持发布/订阅模式和批量处理模式，并且在分布式系统中具有很高的可伸缩性和可靠性。
  • Redis： Redis 是一款高性能、内存型的键值存储系统，也可以用作消息队列中间件。Redis 提供了发布/订阅模式和队列模式，并且支持多种数据结构和数据持久化选项。
  • Apache Pulsar： Pulsar 是一款多租户、高可用性的分布式消息队列和流处理平台。它支持发布/订阅模式和队列模式，并且可以与其他数据处理系统集成。

消息队列选型

  • 开源产品
  • 有一定的社区活跃度
  • 与生态有良好的结合
  • 消息可靠传递
  • 高可用
  • 性能

如果您需要处理大量的实时数据流，并且需要高吞吐量和低延迟，那么 Kafka 可能是一个很好的选择，并且在分布式系统中具有很高的可伸缩性和可靠性。

如果应用程序需要严格的消息传递顺序，并且需要较高的可靠性和可用性，那么 RocketMQ 可能是一个很好的选择。适用于金融、电商等对可靠性和消息顺序要求较高的应用场景。

如果需要处理多租户、多语言和多协议的消息，并且需要灵活地进行数据处理和流式计算，那么 Apache Pulsar 可能是一个很好的选择。

kafka vs RocketMQ

相同点：

  • 都是发布-订阅模式
  • 高性能

不同点：

  • 底层实现不同：Kafka使用持久化日志来存储消息，而RocketMQ使用类似于标准JDBC的存储方式存储消息。
  • 语言不同：Kafka是用Scala语言编写；RocketMQ是用Java语言编写。
  • 集群管理不同：Kafka使用Zookeeper来管理集群状态，而RocketMQ使用Name Server来管理集群状态。
  • 应用场景不同：Kafka更适合处理海量数据的传输和分析，例如日志处理、监控系统、报警系统等。RocketMQ更适合处理高并发、高可靠性的应用场景，例如电商、金融等需要强顺序性的场景。

消息模型

队列模型

  • 队列：先进先出
  • 弊端：一个消息只能被其中一个消费者消费
发布-订阅模型

  • 包含模块
    • 发布者
    • 订阅者（通过消费者组实现）
    • 主题/Topic
      • queue队列（rocketmq）/分区partition（kafka）
      • 消费位置
    • 消费者组
  • 队列的作用
    • 为了保证消息的有序性，在一条消息被成功消费之前，下一条消息不能被消费。也就是说，每个主题在任意时刻，至多只能有一个消费者实例在进行消费。为了解决这个问题，提升消费性能，引入队列的概念。
    • 每个主题包含多个队列，通过多个队列来实现多实例并行生产和消费。
    • 只在队列上保证消息的有序性，主题层面不保证
  • 消费者组
    • 解决同一个消息多个业务消费的问题
    • 每个消费组都消费主题中一份完整的消息，不同消费组间消费进度不受影响
    • 消费组中包含多个消费者，同一个组内的消费者是竞争消费的关系，每个消费者负责消费组内的一部分消息。
  • 消费位置
    • 由于消息需要被不同的消费组进行多次消费，所以消费完的消息并不会立即删除，所以消息队列为每个消费组在每个队列上维护了一个消费位置。

消息队列常见场景

  • 如何实现消息有序
  • 如何保证消息不丢失
  • 如何保证消息只消费一次（幂等）
  • 消息积压如何处理
  • 分布式事务

kafka

kafka是什么

Kafka 起初是由 Linkedin 公司采用 Scala 语言开发的一个多分区、多副本且基于ZooKeeper协调的分布式消息系统，现己被捐献给 Apache 基金会。目前 Kafka 已经定位为一个分布式流式处理平台，它以高吞吐、可持久化、可水平扩展、支持流数据处理等多种特性而被广泛使用。

kafka的应用场景

  • 日志收集
  • 消息系统
  • 大数据实时计算

kafka系统架构

  • Producter
    • 通过push模式向Broker发送消息
  • Broker
    • 存储消息的服务器。Broker节点数量越多，集群吞吐率越高
  • Group
    • 通过pull模式从Broker订阅并消费消息
  • Zookeeper
    • 管理集群的配置、选举leader分区，并在Group发生变化时，进行负载均衡

kafka是如何保证消息不丢失的

  • Producter端发送到Broker
    • producter端发送消息时，有ack消息确认机制。只有当kafka中的若干个broker端收到消息并写入日志文件后，才会返回成功。
    • ack=0，发送后就认为成功；ack=1，leader接收成功就表示发送成功；ack=-1或all，全部接收成功才算成功。
    • ISR机制。能和leader副本保持同步的follower副本组成的集合进行同步。动态调整，一旦follower出现故障或延迟，就会被剔除ISR。
    • kafka采用的就是一种完全同步的方案，而ISR是基于完全同步的一种优化机制。
    • 读写都是由leader处理，follower只是作备份功能，不对外提供服务。
  • Broker同步消息并持久化
    • Broker 集群接收到数据后会将数据进行持久化存储到磁盘，为了提高吞吐量和性能，采用的是「异步批量刷盘的策略」。
    • 接收到的数据会先存到PageCache里
    • 多分区多副本机制，最大限度的保证数据不丢失
  • Consumer从Broker拉取消息并消费
    • commit机制。先处理消息，再提交offset。
    • 消息幂等处理，避免服务宕机offset未提交成功重复消费的问题

幂等性

kafka多副本机制

用途：保证kafka的高可用

kafka采用领导者副本机制

  • 副本分为两类：领导者副本和追随者副本。每个分区在创建的时候都要选举一个副本，成为领导者副本，其余的副本则为追随者副本。
  • follower副本不对外提供业务。唯一的任务是从leader副本异步拉取消息并同步。
  • leader副本挂了后，开启leader选举，从follower中选举一个leader。

该机制的好处

  • read your wirtes。实时写，实时读。
  • 实现单调读。始终可读。

kafka如何实现高吞吐和高性能的

  • 页缓存技术
    • kafka基于操作系统的页缓存实现文件写入。页缓存叫Page Cache，是操作系统在内存中给磁盘文件建立的缓存。写入文件的时候，数据写入Page Cache就返回了。属于内存的操作，所以速度很快。而且一般情况写入队列就会立即被消费，所以可以直接命中缓存，读取速度也非常快。
  • 磁盘顺序写
    • 写数据的时候，是将数据追加到文件的末尾，不是在文件的随机位置修改数据。对于磁盘来说，顺序读写的性能要远远高于随机读写。所以，可以大大提高磁盘IO性能。
  • 零拷贝技术
  • 批消息概念
    • 整个过程中都是对一批消息进行处理
  • 数据压缩
零拷贝技术
  • 消费部分。从kafka的磁盘文件里读取某条数据发送给下游的消费者。
  • 正常过程是，将数据从磁盘读取到page cache，再从page cache拷贝到应用程序的内存空间，再拷贝到操作系统的socket缓冲区。要进行多次数据拷贝，比较耗费性能。
  • kafka引入零拷贝技术，将page cache中的数据直接发送到网卡，跳过了中间两次拷贝数据的步骤。其实就是只传递消息的指针，消费者可以直接从page cache中读取数据。

kafka文件存储机制

  • Broker
    • 一个Kafka节点就是一个broker，多个broker可以组成一个Kafka集群
  • Topic
    • 主题，一类消息
  • Partition
    • topic物理上的分组，一个topic可以分为多个partition，每个partition是一个有序队列
  • Segment
    • log：数据文件
    • index：索引文件
    • timeindex：时间索引文件
  • offset
物理存储
  • 数据存储目录
    • 每个partition为一个目录，命名：topic名称+分区序号
      • log、index、timeindex三个文件
      • partion全局的第一个segment从0开始，后续每个segment文件名为上一个segment文件最后一条消息的offset值。
如何根据 offset 找到对应的 Message
  1. 根据offer找到对应的segment文件。可以直接根据segment的文件名查找。
  2. 从segment中获取的对应的offer数据。根据索引定位。index文件存储了offset（消息在文件中的偏移量）和position（消息的物理偏移地址），去log文件中找对应的message信息。（index文件中并没有为数据文件中的每条消息都建立索引，而是采用了稀疏存储的方式，每隔一定字节的数据建立一条索引。避免了索引文件占用过多的空间，从而可以将索引文件保留在内存中。）

RocketMQ

RocketMQ的架构

  • nameServer。无状态集群，管理Broker集群，维护topic信息
  • broker。负责存储消息，转发消息。
  • producter。负责发送消息。
    • 三种发送消息方式。同步发送，异步发送，单向发送
  • consumer。负责消费消息。
    • 两种消费模式，PUSH和PULL。

RocketMQ的工作流程

  1. broker在启动的时候向nameServer注册，并维持长连接，每30s发送一次心跳
  2. producter与nameServer集群随机一个节点建立长连接，定期获取topic信息；根据负载均衡算法，选择一个broker建立长连接，推送消息。
  3. consumer与nameServer集群随机一个节点建立长连接，定期获取topic信息；从broker建立长连接，拉取消息。
RocketMQ的长轮询

所谓的长轮询，就是Consumer 拉取消息，如果对应的 Queue 如果没有数据，Broker 不会立即返回，而是把 PullReuqest hold起来，等待 queue 有了消息后，或者长轮询阻塞时间到了，再重新处理该 queue 上的所有 PullRequest。

延时消息原理

临时存储+定时任务

  • 发送到相应时间段的message queue中
  • 通过定时任务轮询队列，到期投递到目标topic中。

死信队列

当一条消息初次消费失败，消息队列 RocketMQ 会自动进行消息重试；达到最大重试次数后，若消费依然失败，则表明消费者在正常情况下无法正确地消费该消息，此时，消息队列 RocketMQ 不会立刻将消息丢弃，而是将其发送到该消费者对应的特殊队列中，该特殊队列称为死信队列。

RocketMQ的高可用

  • Producer只能向Master角色的Broker写入消息，Cosumer可以从Master和Slave角色的Broker读取消息。
